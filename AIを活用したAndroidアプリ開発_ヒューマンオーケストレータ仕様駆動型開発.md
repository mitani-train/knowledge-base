# AIを活用したAndroidアプリ開発

## ～ヒューマンオーケストレータ仕様駆動型開発～

### 概要

本手法は、人間が仕様の作成と品質判断を担当し、AIが実装を担当する開発手法です。

人間はコードを書くことを主目的とせず、アプリの仕様を定義し、その仕様に基づいて複数のAIをオーケストレーションします。

仕様を中心に開発を進めるため、AIや開発ツールが変わっても再利用しやすく、継続的な改善が行いやすいことが特徴です。

### 「オーケストレーション」とは

オーケストレーションとは、人間が指揮者となり、複数のAIをそれぞれの役割に分割して操作することを指します。

オーケストラの指揮者が各楽器の演奏者に指示を出し、全体として一つの楽曲を完成させるように、本手法では人間がChatGPT・Claude・Codexといった複数のAIに対してそれぞれ異なる役割を与え、組み合わせて一つのアプリを完成させます。

人間自身がコードを書くのではなく、「どのAIに何をさせるか」を決定し、各AIの出力を次の工程へつなげていく立場を担います。

### 「仕様駆動型」とは

仕様駆動型とは、AIの出力結果を一定の品質・方向性に保つために、あらかじめ仕様を策定し、その仕様に沿ってAIに開発させる手法を指します。

AIへの指示が曖昧な場合、同じ依頼であっても出力結果がそのたびに変わってしまうことがあります。仕様書という共通の基準を用意し、各AI（ChatGPT・Claude・Codex）がこの仕様書を参照しながら作業することで、

* 工程をまたいでも判断基準がぶれない
* AIや担当者が変わっても仕様書を引き継げる
* 改善のたびに仕様書を更新すれば、過去の意思決定が記録として残る

といった効果が得られます。仕様書が開発全体の「設計図」であり、コードはその設計図から生成される派生物という位置づけになります。

### なぜ複数のAIサービスを使い分けるのか

本手法では、壁打ち・仕様書作成を担当するAI（ChatGPT）と、実装プロンプト生成を担当するAI（Claude）を、意図的に別のサービスとして分離しています。この2つの役割は入れ替えても構いませんが、**同一サービス内で分担しないこと**が重要です。

同一サービス内で新規チャットを分けていても、サービス側の仕組みによってチャット内容や文脈が共有される場合があります。この場合、本来は独立しているはずの「壁打ち・要件整理の工程」と「実装プロンプト生成の工程」が暗黙的に影響し合ってしまい、役割分担が崩れる可能性があります。

加えて、AIは与えられたタスクに対して近道（ショートカット）的な振る舞いをとりやすい傾向があります。前工程の文脈情報にアクセスできる状態だと、AIが本来評価されるべき基準を経由せず、文脈から答えを推測してしまう、いわゆる**shortcut learning（近道学習）や文脈汚染**に近い挙動が起きやすくなります。

そのため本手法では、

* 壁打ち・仕様書作成 → ChatGPT
* 仕様書解析・実装プロンプト生成 → Claude

のように、**役割ごとに異なるAIサービスを割り当て、工程間でチャット文脈を共有させない**ことを設計上の原則としています。これにより、各AIは前工程の文脈に頼らず、毎回「仕様書」という共通の入力のみを根拠に出力することになり、仕様駆動型開発の前提である「出力の一貫性」が保たれます。

# 本手法における注意点

## 機密情報の取り扱いについて

本手法では、ChatGPT・Claude・Codexといったインターネット上のAIサービスを利用します。

これらのサービスに入力した内容は、サービスによって学習データとして利用される場合があります。

そのため、以下のような情報は仕様書やプロンプトに含めないことを徹底してください。

* 個人情報（氏名、連絡先、顧客情報など）
* 社外秘の業務情報・社内システムの詳細
* APIキーやパスワードなどの認証情報
* 公開前の事業計画・未発表の機能情報
* 実データ（本番データベースの内容など）

仕様書作成やエラー修正の際は、実データではなくダミーデータ・架空のユーザー名や数値を用いるようにしてください。

各AIサービスには学習利用に関する設定（オプトアウト機能など）が用意されている場合がありますが、設定内容はサービスやプランによって異なり、変更される可能性もあるため、設定への依存だけに頼らず、入力する情報そのものを管理することが基本方針となります。

## 使用ツール

### ChatGPT

* アイデアの壁打ち
* 要件整理
* 仕様書作成

### Claude

* 仕様書の解析
* Codex向け実装プロンプト生成
* ビルドエラー・Logcatの解析と修正方針の提示

### Codex

* コード生成
* コード修正

### Android Studio

* ビルド
* 実行
* 動作確認
* エラー解析

### 人間（ヒューマンオーケストレータ）

* 仕様決定
* 品質評価
* 改善方針決定

# 開発手順

## 1. 環境構築

Android StudioとCodexをインストールします。

Android Studioで新規プロジェクトを作成します。

作成したプロジェクトのルートフォルダをCodexの作業フォルダとして設定します。

---

## 2. アプリのアイデアを整理する

まずChatGPTを使い、作りたいアプリの内容を壁打ちします。

以下のような内容を整理します。

* 誰が利用するのか
* 何を解決するアプリなのか
* どのような機能が必要か
* どのような画面が必要か
* どのようなデータを扱うのか

この段階では実装方法ではなく、アプリの目的や利用イメージを明確にすることが重要です。

---

## 3. 仕様書を作成する

壁打ち結果をChatGPTへ入力し、仕様書を作成します。

仕様書には以下の内容を含めます。

* アプリ概要
* 画面一覧
* 機能一覧
* 画面ごとの動作
* 入力項目
* データ保存方法
* 制約事項

仕様書は本開発手法における中心成果物です。

---

## 4. ClaudeでCodex用プロンプトを作成する

作成した仕様書をClaudeへ入力します。

Claudeには、

「この仕様書を実装するためのCodex用プロンプトを作成してください」

と依頼します。

Claudeは仕様書を解析し、Codexが実装しやすい形へ変換します。

この工程では設計を行うのではなく、仕様書を実装指示へ翻訳することが目的です。

---

## 5. Codexで実装する

Claudeが生成したプロンプトをCodexへ入力します。

Codexはプロジェクト内のコードを作成・修正します。

大きな機能を一度に実装するのではなく、

* 画面単位
* 機能単位
* 修正単位

で進めると成功率が高くなります。

---

## 6. Android Studioでビルドする

Codexによる実装後、Android Studioでビルドを実行します。

エラーが発生した場合は内容を確認します。

---

## 7. エラーを修正する

ビルドエラーや実行エラーが発生した場合は、

* エラーログ
* Logcat
* スタックトレース

をClaudeへ入力し、原因分析と修正方針を確認します。

Claudeが提示した修正内容をもとに、Codexへ修正指示を出します。

修正後は再度ビルドを実施します。

---

## 8. 動作確認を行う

エミュレータまたは実機で動作確認を行います。

確認項目の例

* アプリがクラッシュしないか
* 仕様通り動作するか
* UIに問題がないか
* データが正常に保存されるか

---

## 9. 改善サイクルを回す

動作確認の結果から、

* 不満点
* 改善点
* 追加機能

を整理します。

その内容を再度ChatGPTへ入力し、壁打ちを行います。

その後、

仕様書更新
↓
Claude
↓
Codex
↓
Android Studio
↓
動作確認

を繰り返します。

---

# 本手法の考え方

従来の開発では、人間が要件定義から実装まで担当していました。

本手法では、

人間
→ 仕様を作る

AI
→ 仕様を実装する

という責任分担を行います。

人間は仕様の品質に責任を持ち、AIは実装を担当します。

つまり成果物の中心はコードではなく仕様書です。

仕様書を基準として継続的に改善を行うことで、AIを活用した効率的なアプリ開発を実現します。

# 付録：プロンプト例

具体例として、「発表練習支援アプリ」を開発する場合のプロンプト例を示します。

## 1. ChatGPTで壁打ちする

### 入力例

```
私はAndroidアプリを作りたいです。

アイデアは「発表練習支援アプリ」です。

発表音声を録音し、
・話す速度
・間の長さ
・フィラー（えー、あのー）
を分析したいです。

以下の観点で壁打ちしてください。

* 想定ユーザー
* 必要な機能
* MVPとして最低限必要な機能
* 将来的な拡張案
```

### 出力イメージ

* 想定ユーザー
* 解決したい課題
* MVP機能一覧
* 将来機能一覧

を整理した結果を得る

---

## 2. ChatGPTで仕様書を作成する

### 入力例

```
以下の内容からAndroidアプリの仕様書を作成してください。

【壁打ち結果】
（壁打ち結果を貼り付け）

仕様書には以下を含めてください。

* アプリ概要
* 画面一覧
* 機能一覧
* 画面ごとの動作
* データ保存方法
* 制約事項
```

### 出力イメージ

Markdown形式の仕様書

---

## 3. ClaudeでCodex用プロンプトを作成する

### 入力例

```
以下の仕様書を実装するためのCodex用プロンプトを作成してください。

条件

* Android
* Kotlin
* Jetpack Compose
* MVVM構成
* Room Database使用
* 差分実装
* 既存コードを壊さない

Codexがそのまま実行できる形で出力してください。

【仕様書】
（仕様書を貼り付け）
```

### 出力イメージ

Codexに投入するための詳細な実装指示

---

## 4. Codexで実装する

### 入力例

```
以下の内容を実装してください。

【対象】
発表履歴一覧画面

【要件】
* Jetpack Compose使用
* Roomから履歴取得
* LazyColumnで表示
* 日時順にソート
* MVVM構成

必要なファイルをすべて修正してください。
ビルド可能な状態で実装してください。
```

### 出力イメージ

* ViewModel作成
* Repository作成
* Compose画面作成
* Room連携

---

## 5. ビルドエラー修正

### 入力例

```
以下のビルドエラーを修正してください。

【エラー】
（エラーログ貼り付け）

原因分析と修正内容を説明した後、
修正版コードを出力してください。
```

---

## 6. 動作不具合修正

### 入力例

```
以下の不具合を修正してください。

現象
録音ボタンを押すとアプリがクラッシュする。

Logcat
（Logcat貼り付け）

考えられる原因と修正案を提示してください。
```

---

## 7. 機能追加

### 入力例

```
現在の仕様書に以下の機能を追加したいです。

* AIによる発表評価
* 話す速度のグラフ表示

仕様書へ追記してください。
変更点のみ出力してください。
```

---

## 8. 改善サイクル

### 入力例

```
現在のアプリを利用して感じた課題です。

* 操作が分かりにくい
* 分析結果が見づらい
* グラフが小さい

UI/UX改善案を提案してください。
優先順位付きで出力してください。
```
